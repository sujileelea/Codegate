## 1. 종합 판정

**아키텍처 수준 통제는 설계 의도대로 실구현되어 강하다** — 5스키마 분리, per-user DEK 엔벨로프 암호화, crypto-shredding, 의존성 계층 인가, 감사 로그, LLM 화이트리스트 스크러빙·비진단 필터, 멱등키 스코프, 입력 크기 상한, 로그 마스킹이 모두 코드로 확인됐다.

**배포 전 조치(§3)와 운영 인프라 통제(§6)는 조치·검증 완료 상태다.** 남은 것은 아래로 좁혀졌다:

1. **배포 전 조치 잔여 2건** — 토큰 폐기 실활성화(옵션 구현 완료, Admin SDK 자격증명 제공 대기)와 시크릿 관리 고도화(SSM SecureString 이관 완료 — 앱 코드의 KMS 직접 로드·키 로테이션 절차 미완). 나머지(note_text 암호화·레이트리밋 폴백·요청 경로 LLM 통제·non-root)는 완료(§3).
2. **신규 기능 게이트** — 음성 일기·음성 응원 편지·구독·가족 선물은 §4의 전제 통제가 구현되기 전에는 해당 기능을 출시하지 않는다.
3. **실결제 전환 게이트** — 충전·출금·기프티콘의 실결제 전환은 자금 흐름 남용 방어(§4.7 M-1~M-8: 본인 명의 출금·현금성 한도·냉각 기간·FDS·전금법/AML 검토)가 갖춰지기 전에는 하지 않는다.

## 3. 배포 전 필수 조치 — 조치 결과

1. **note_text DEK 암호화 승격 — 완료**: 평문 컬럼 제거, 시니어 DEK 암호문 컬럼(`note_text_enc`) 전환 + 마이그레이션 적용. 음성 일기 전사(§4.1 V-3)는 이 경로를 재사용한다.
2. **레이트리밋 fail-closed 전환 — 완료**: Redis 장애 시 프로세스-로컬 고정 윈도 카운터로 계속 집계(무제한 통과 경로 제거). 전 표면(auth/session·families/join·reports:generate) 공통 적용, 단위 테스트 포함.
3. **`check_revoked` — 구현 완료·리스크 수용(오너 결정)**: GCP 조직 정책상 서비스 계정 키 생성 불가 확인 → 미활성 수용. 근거: 토큰 수명 1h 상한 + `status='active'` 즉시 차단. 활성화 경로는 Workload Identity 제휴(키리스 — 조직 정책 부합)로 확보되어 있으며 정식 서비스 전 전환 검토.
4. **`/reports:generate` 요청 경로 LLM 호출 — 완료(예외 명문화)**: 동기 방식 유지(API 계약 §9와 일치)하되 가족당 3회/10분 레이트리밋 + 전역 동시 실행 상한(2, 초과 429) + 프로바이더 타임아웃으로 통제. "요청 경로 미호출" 원칙의 유일한 명시적 예외.
5. **Secret Manager 이관 — 부분 완료**: 마스터 키·HMAC 키·내부 토큰·워커 비밀번호·DB 비밀번호의 원본을 SSM SecureString으로 이관(서버 env는 권한 제한된 운영 사본). 잔여: 앱 코드의 KMS/SSM 직접 로드, DEK 재랩(키 로테이션) 절차 문서화.
6. 잔여 P1(SECURITY.md §5): Dockerfile non-root **완료**(비루트 uid 실행 검증). Redis는 컨테이너 내부 네트워크 전용(외부 포트 미공개)으로 노출 표면 없음 — prod AUTH는 외부 노출 시점에 적용. 미착수: Sentry, 일반 쓰기 API 레이트리밋 확대, access_log·ai_jobs 보존 정책(TTL/파티셔닝).

## 4. 신규 기능 보안 게이트 — 이 통제 없이 해당 기능 출시 금지

### 4.1 똑똑이 음성 일기 (서비스 최대 민감 기능)

시니어의 자유 발화를 수집·분석하는 기능이므로, 아래 8개 통제가 **전부** 구현·검증되기 전에는 출시하지 않는다(부분 출시 금지 — 예: 보존·삭제 없이 녹음 기능만 먼저 여는 것 금지).

| # | 통제 | 기준 |
|---|---|---|
| V-1 | 음성 원본 미수집 | on-device STT만 사용, **오디오 업로드 API 자체를 만들지 않는다**. 서버·스토리지에 오디오가 존재하면 그 자체가 보안 사고로 취급 |
| V-2 | 별도 명시 동의 | `voice_diary` opt-in(§1.7) — 온보딩 필수 동의에 포함 금지, 시니어 본인 기기에서 큰 글씨 설명("목소리는 저장되지 않아요·이야기 내용만 기록돼요·가족에게 보여줄지는 내가 결정해요") 후 동의. 철회 = 전사·파생 지표 전체 삭제(비가역 안내) |
| V-3 | 전사 암호화 | DEK 암호화 컬럼 저장(note_text 승격과 동일 경로), activity 스키마(이름·연락처 없음) |
| V-4 | 공개 제어는 시니어 본인 | 전사 원문은 **기본 비공개**(guardian 403). 공개 여부는 시니어 본인의 공유 설정(`voice_diary_share_with_family` — 본인만 변경 가능, guardian 변경 시도 403)으로만 전환되며, 켠 경우 guardian 열람마다 감사 로그, 끄는 즉시 접근 회수. 꺼진 상태에서 가족에게는 참여 여부·정서 신호·발화 특징 **변화 요약**만(비진단 문구). 이 경계는 서버 인가 의존성으로 강제 |
| V-5 | 외부 LLM 전송 조건 | PII 스크러빙(이름·연락처·주소·기관·지명) + 가명 ID + 프롬프트 원문 미저장. **무료 티어(입력을 학습에 쓸 수 있는 약관)로는 전사를 외부 전송하지 않는다** — 유료/미학습 보장 약관 확인 전까지 통계 로직 지표만 운영(API_SPEC §9) |
| V-6 | 보존·파기 | **확정(오너 결정)**: 유효 티어 free = 30일 후 원문 파기(파생 지표만 보존), premium 상속 = 영구 보존. 일 배치가 파기 수행(구현·배포됨). 본인 삭제 시 파생 지표 동반 삭제 |
| V-7 | 비진단 표현 | 발화 특징(반복·개연성·단어 회상 신호)은 "변화 참고 신호"로만 노출 — 진단 어휘 필터 + 리포트 사람 검수. "인지 저하·치매 의심" 류 표현은 코드 레벨 금지어 |
| V-8 | UX 안전장치 | 녹음 중 대형 표시·즉시 중단, 전사 미리보기 후 시니어가 직접 전송(자동 전송 금지), 마이크·음성인식 권한 거부 시 기능 비활성 안내 |

### 4.2 프리미엄 구독

- 게이트는 **서버가 402로 강제** — 클라이언트 잠금 UI만으로 배포 금지. 티어 판정은 단일 entitlement 의존성으로 구현(라우터별 자체 판정 금지).
- StoreKit2 트랜잭션은 서버 검증, 카드 정보 미수집(트랜잭션 ID만). `demo` 상태와 실결제 상태를 스키마에서 구분 — 데모 티어로 실결제 혜택이 열리는 경로가 없어야 한다.
- 시니어 본인의 자기 데이터 열람은 티어와 무관하게 항상 허용(부모 데이터를 볼모로 한 과금 금지). **시니어 화면에는 결제 유도를 넣지 않는다**(S-15 잠금 안내도 "자녀분 구독으로 열려요" 톤 — 다크패턴 방지).
- 실결제 전환 시 App Store Server Notification(갱신·환불) 연동 + 환불 시 티어 즉시 회수.
- **무료 체험(trial — 도입 확정) 남용 방지**: 중복 체험 차단 기준(phone_hmac 기반 계정당/가족당 1회 등)을 A-7 ④ 확정과 함께 서버에서 강제 — 탈퇴 후 재가입으로 체험을 반복 수령하는 경로가 없어야 한다.

### 4.3 음성 응원 편지 (자녀 음성의 서버 저장 — 음성 일기와 다른 트랙)

자녀 음성은 **전달 콘텐츠**이므로 서버 저장이 불가피하다(시니어 음성 일기의 "미수집" 원칙과 혼동 금지). 저장하는 대신 아래 통제로 표면을 좁힌다.

| # | 통제 | 기준 |
|---|---|---|
| L-1 | 저장 최소화 | 오디오 AAC ≤60초·5MB(형식·크기 서버 검증), 전사 ≤1,000자. 오브젝트 스토리지 **암호화 at rest**, 전사는 DEK 암호화 |
| L-2 | 자동 파기 | 보존 상태는 **24h 또는 permanent 둘 중 하나 — 발송 시점의 발신자 티어로 확정, 이후 불변**(스키마에 명시적으로 저장, 구독 상태를 재조회해 판정하지 않는다). 24h 발신분은 스토리지 수명주기 + DB 파기 배치 이중화로 파기, 사후 보관 전환 경로 없음. permanent 발신분은 구독 해지와 무관하게 유지 |
| L-3 | 분석 투입 금지 | 응원 오디오·전사는 발화 특징·정서 분석·LLM 등 **프로파일링 목적의 어떤 파이프라인에도 넣지 않는다**(코드 리뷰 체크 항목). 발화 분석은 시니어 본인 동의 하의 음성 일기 전용. **유일한 예외는 L-7 안전 필터** |
| L-4 | 접근 경계 | 수신 시니어 + 발신자만. 오디오는 **15분 서명 URL**로만 제공, URL 로그 기록 금지, 가족 해제 시 접근 즉시 회수 |
| L-5 | 익명 정책 | `anonymous`는 표시상 익명 — **서버는 발신자를 항상 기록**(남용·괴롭힘 대응, 법적 요청 대비). 개인정보처리방침에 명시 |
| L-6 | 남용 방지 | guardian당 시니어별 3회/일 레이트리밋, 같은 가족만 발송 가능 |
| L-7 | 안전 필터(모더레이션) | 발송 시점 1회, **서버 내 코드**(금칙어·패턴 사전: 욕설·모욕·비하 / 위협·협박 / 위험 행위 조장 / 금전 요구·사기 의심)로 `transcript`만 검사 — **외부 AI 미전송·오디오 미검사**. 감지 시 400 거부(저장 없음), 필터 로그는 통과/차단 여부만(문구 원문 미기록). 클라 로컬 사전 1차 검사는 UX 보조일 뿐 서버 판정이 정본. 잔여 한계: 전사 기반 검사라 오디오-전사 불일치 우회 가능성 존재(수용) |

### 4.4 상점 확장·제휴·가족 선물하기

- 실발급·실물 배송은 실결제 범위 확정 전까지 전 상품 `fulfillment_type=demo` 고정(서버에서 강제).
- `partner` 상품은 광고성 표기 배지 필수(표시광고법 관점), 외부 링크 이동 시 사전 고지. 제휴사에 개인정보를 전달하는 설계는 두지 않는다(주문 발생 시점에 별도 동의 설계 필요 — 현 단계 범위 외).
- **가족 선물하기**: 수령인은 같은 가족 구성원만(서버 검증, 비가족 403). 재원 차감은 서버 확정(시니어 = 본인 지갑 / guardian = 가족 충전 잔액) + 멱등키. **실물 배송 주소는 수집하지 않는다** — demo 이행 한정, 실배송 도입 시 주소 수집 보안 검토 선행(§1.9 신규 필드 원칙).

### 4.5 형제자매 초대·충전 기여 표시

- 기존 권한 모델 재사용이라 새 표면 없음. 충전 내역(`charged_by`)은 가족 구성원 한정 조회 — 외부인 403은 기존 의존성으로 커버됨을 테스트로 확인.

### 4.6 케어 푸시·아바타 (경량 통제)

- **케어 푸시**: ⓐ 푸시 페이로드는 관계 호칭 수준만("어머님의 활동이 3일째 없어요") — 실명·활동 상세·점수 포함 금지(잠금화면 노출 표면 최소화, API_SPEC §10) ⓑ 발송 필터는 서버의 notification-settings가 정본(클라 필터 금지) ⓒ 디바이스 토큰은 로그아웃·탈퇴 시 삭제 + 무효 토큰 자동 정리 ⓓ 케어 알림 생성·푸시는 **티어 무관**(안전 기능에 과금 게이트 금지 — §4.2와 동일 원칙) ⓔ 재알림은 3일 간격(알림 피로·불안 조장 방지).
- **아바타**: 저장 값은 머리·옷 코드 2개뿐(PII 없음 — 평문 허용). 서버가 카탈로그 존재·소유 여부를 검증(비소유 유료 아이템 400) — 클라이언트 신고 값 불신 원칙 동일 적용. 새 보안 표면 없음.

### 4.7 자금 흐름 남용 방어 (자금세탁·보이스피싱·탈세 — 실결제 전환의 전제)

앱에 부모-자식 관계의 공식 증명 장치가 없으므로, "충전(A의 카드) → 시니어 계정 수취 → 출금·기프티콘 현금화(B의 손)"의 자금 이동 채널로 악용될 수 있다. 위협 시나리오: ① 자금세탁·탈세(차명 시니어 계정으로 자금 이전) ② 보이스피싱(피해자에게 충전 유도, 사기범 통제 계정이 수취) ③ 계정 탈취(시니어 계정 접근 후 출금 계좌 변경). **실결제·실출금·실발급이 전부 데모인 현재는 리스크가 없으며, 아래 통제는 실결제 전환(A-7 ②③)의 전제 조건이다.**

구조적 방어(현 설계에 이미 존재): 포인트 직접 이체 불가 — 충전 잔액은 미션 보상으로만 시니어에게 흐르고 주간 상한(weekly_cap)이 유속을 제한 / 충전은 IAP 경유(수수료 구조상 세탁 비경제적 + 카드 발급사 KYC) / 실명·전화번호 수집 + `phone_hmac` 중복 탐지. **잔여 허점은 2개**: 현금성 선물(기프티콘 깡 — 주간 상한을 우회하는 유일한 대량 경로)과 타인 명의 출금 계좌.

| # | 통제 | 기준 |
|---|---|---|
| M-1 | 본인 명의 출금 | 출금 계좌는 **시니어 본인 명의만**(1원 인증 또는 계좌실명조회로 가입 실명과 대조). 이 검증 없이 실출금 기능 출시 금지 — 관계 증명 없이도 "돈이 계정 명의자 밖으로 나가지 않는다"를 성립시키는 핵심 통제 |
| M-2 | 환불 경로 고정 | 미사용 충전 포인트 환불은 **원결제수단으로만**(App Store 환불) — 제3자 계좌 환불 경로를 만들지 않는다 |
| M-3 | 현금성 유속 제한 | 충전 월 한도(가족당·인당), 출금 일/월 한도, **현금성 선물(gifticon·health·partner) 월 한도는 소액**으로 별도 설정 — 깡 경로의 경제성 제거. 비현금성(deco·theme·widget)은 제외. 수치는 A-7 ⑩ |
| M-4 | 냉각 기간 | 신규 가족 연결 후 일정 기간(후보 72시간~7일) 출금·현금성 선물 금지, 출금 계좌 변경 후 48시간 출금 보류 — 보이스피싱의 속도전 차단 |
| M-5 | 이상 거래 탐지(FDS) | 통계 로직 일 배치(기존 배치 인프라 재사용) 룰: 가입 직후 고액 충전 / 충전→즉시 현금성 소비 / 한 계정의 다수 가족 연결 / 동일 기기의 시니어+guardian 겸직 / **미션 참여 없는 지갑**(실사용 시니어는 활동 데이터가 쌓임 — 이 서비스 고유의 강한 신호). 감지 시 거래 보류·한도 축소 + 확인 절차. 활동 정상 계정은 영향 없음 |
| M-6 | 다계정 탐지 | `phone_hmac` 중복 가입 차단(기존), 기기 식별 기반 계정 연결 탐지 |
| M-7 | 자금 흐름 감사 추적 | 충전→보상→출금·선물 전 구간이 append-only 원장으로 연결(기존) — 사후 자금 흐름 재구성 가능 상태 유지 |
| M-8 | 법제 정합 | 실출금은 **선불전자지급수단 발행 해당 여부 검토 + 전자금융거래법 등록·AML/고객확인 의무 검토가 선행**돼야 개발 착수 가능(A-7 ③). 공식 관계 증명은 기본 미요구 — 고한도 필요 가족만 **선택적 관계 인증으로 한도 상향**하는 티어 구조 |

## 6. 운영·인프라 보안 원칙

- **시크릿**: 저장소에는 평문 비밀 금지 — 원본은 SSM SecureString, 서버 env는 권한 제한 사본, CI gitleaks 유지, `.gitignore`의 자격증명 패턴 유지. 접속 정보(도메인·리소스 식별자)는 git 밖 워크스페이스 로컬 파일에만 둔다.
- **운영 AWS 인프라**(EC2+RDS, prod 모드) — 적용·검증된 상태: 저장 암호화(RDS StorageEncrypted, EC2 루트 볼륨 암호화, 계정 기본 EBS 암호화), 전송 암호화(도메인 TLS 종단 — 공인 인증서 자동 갱신, 앱→RDS TLS), 인바운드 최소 개방(SSH=운영자 IP/32, 80·443만 공개, API 포트 직접 노출 폐쇄), RDS 비공개+삭제 방지, prod에서 API 문서(/docs·/openapi.json) 비공개, 컨테이너 비루트 실행, 콘솔 계정 MFA. 미암호화 구세대 리소스(DB·볼륨·스냅샷)는 폐기 완료.
- **오브젝트 스토리지**(음성 응원 편지 도입 시): 버킷 전체 비공개(퍼블릭 액세스 차단), 서버 측 암호화, 24시간 수명주기 규칙, 서명 URL 외 접근 경로 없음, 버킷 정책으로 애플리케이션 롤만 허용.
- **배포 게이트**: prod 배포·인프라 변경 전 감사 체크(§3 목록 + SECURITY.md P1)를 통과하지 않으면 배포하지 않는다.
- **관측**: Sentry(오류)·LLM 호출 실패/비용 대시보드 — 미착수 항목이며 P1에 포함.
