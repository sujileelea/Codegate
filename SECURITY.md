# 똑똑용돈 백엔드 보안 점검 보고서

> ⚠️ **문서 성격 재정의**: 이 문서는 특정 구현의 실사 결과가 아니라 **목표 보안 설계·구현 체크리스트**다. 백엔드는 아직 미구현이며, 본문의 "구현·검증됨(✅)" 표기와 점검 커밋·일자·등급은 **구현 시 달성해야 할 목표 상태**로 읽는다. 실제 점검 결과는 백엔드 구현 후 §6 이력에 새로 기록한다.
> 정책 확정 반영 필요: 수집 항목 확대(실명·**이메일**·전화번호·성별·**생년월일** — [API_SPEC.md](./API_SPEC.md) §2가 정본, §1 인벤토리의 "생일 대신 연도만" 표기 갱신 대상) · 걸음 기능 제거(걸음 수 미수집) · 로그인 Apple+카카오(OIDC).
> 상태 표기: ✅ 구현·검증됨 / ⚠️ 부분 구현·보강 필요 / ❌ 미구현
> 등급 기준: **A** 잔여 리스크 미미 / **B** 핵심 통제 구현, 보강 여지 / **C** 설계 대비 구현 공백 / **D** 통제 부재

## 1. 수집 데이터 인벤토리 및 보호 필요 데이터 선별

민감도 기준: **상** = 직접 식별 또는 건강 추정 가능 / **중** = 간접 식별·행동 데이터 / **하** = 비식별 운영 데이터

| 데이터 | 저장 위치 | 민감도 | 선별 근거 | 현재 보호 조치 |
|---|---|---|---|---|
| 이름(display_name), 전화번호 | `identity.user_profiles` | **상** | 직접 식별자 | AES-256-GCM 암호화(per-user DEK), 검색은 HMAC 인덱스만 |
| 관계 라벨('어머니' 등) | `identity.family_members` | **상** | 가족관계 정보 | 본인 DEK로 암호화 |
| **인지 수행 데이터** (정답률·반응시간·오류·도메인별 추이) | `activity.*`, `ai.*` | **상** | 축적 시 인지 기능 변화 **추정 가능 → 건강정보에 준하는 취급 필요**. 이 서비스의 최대 민감 자산 | 스키마 물리 분리(이름 부재), UUID 논리 참조, LLM 화이트리스트, 비진단 원칙 |
| 체크인 자유 텍스트(note_text) | `activity.checkins` | **상** | 사용자가 임의 PII 입력 가능(이름·전화 등) | LLM 입력 제외 + 마스킹 유틸. **저장은 평문** ⚠️ |
| 기분·수면·식사·걸음 수 | `activity.checkins` | 중 | 생활 습관 정보 | 스키마 분리, 비식별 |
| 출생연도·성별 | `identity.user_profiles` | 중 | 간접 식별(최소 수집으로 생일 대신 연도만) | identity 격리 |
| 용돈 금액·환산액 | `reward.*` | 중 | 가족 재정 정보(실결제 없음) | 권한 의존성(guardian만 설정), 스키마 분리 |
| 포인트 원장 | `reward.point_ledger` | 중 | 행동 이력 | append-only, 비식별 |
| firebase_uid | `identity.users` | 중 | 외부 시스템 식별자 | identity 격리, 로그 마스킹 |
| AI 리포트·추천 이유 | `ai.family_reports` 등 | 중 | 인지 데이터의 해석 산출물 | 비진단 필터, guardian 디스크 저장 금지 계약 |
| 초대 코드 | `identity.family_invites` | 중 | 가족 접근 토큰 | **HMAC만 저장**(평문은 응답 1회), TTL 24h·1회용 |
| 접근 이력·IP | `audit.access_log` | 중 | 열람 추적 | IP는 HMAC 해시만 저장(원본 미보관) |
| Firebase ID 토큰 | (전송 중만) | **상** | 인증 자격 | 미저장, 로그 마스킹 |
| 미션 카탈로그, 집계 통계 구조 | `activity.missions` 등 | 하 | 비개인 데이터 | — |

**핵심 판단**: 이 서비스에서 가장 보호가 필요한 것은 통념상의 PII(이름·전화)보다 **인지 수행 시계열 데이터**다. 이름은 암호화로 끝나지만, 인지 데이터는 "누구의 것인지 연결되는 순간" 건강 추정 정보가 되므로 **식별자와의 분리 유지가 생명선**이다. 현 설계(스키마 분리 + 논리 참조 + crypto-shredding)는 이 원칙에 부합한다.

## 2. 데이터별 보호 방안 (현행 + 보강)

| 데이터군 | 현행 (구현 확인됨) | 보강 필요 |
|---|---|---|
| 직접 식별자 | 엔벨로프 암호화(DEK per user + 마스터 키 랩), 탈퇴 시 DEK 폐기로 복호화 불능화, HMAC 검색 | 마스터 키를 env → **Secret Manager/KMS 연동**, 키 로테이션 절차 문서화 |
| 인지 수행 데이터 | 스키마 분리, activity에 이름·연락처 부재, 파생 집계만 노출, LLM 화이트리스트 직렬화, 진단 어휘 이중 필터 | **DB 롤 분리 실적용**(§3 D-3), 가명정보 보존 근거(탈퇴 시 비식별 보존) 약관 명시 |
| note_text | LLM 제외, 500자 제한 | **저장 시 패턴 마스킹 적용** 또는 DEK 암호화 저장으로 승격 |
| 초대 코드·내부 토큰 | HMAC 저장, 상수 시간 비교 | join 엔드포인트 레이트리밋, internal 토큰 기본값 사용 시 prod 기동 차단 |
| 접근 이력 | insert-only, IP 해시화 | 보존 기간 정책(월 파티셔닝은 설계에 예정됨) |
