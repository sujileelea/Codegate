# MVP 구현 플랜

> 성격: **MVP로 구현할 것과 발표 확장성 섹션으로 보낼 것의 구분 정본.** 계약 상세는 [API_SPEC.md](./API_SPEC.md), 보안 게이트는 [보안 문서.md](./보안%20문서.md), 발표 서사는 [발표자료.md](./발표자료.md)가 관할한다. 이 문서와 API_SPEC이 다르면 API_SPEC이 우선한다.

## 분리 원칙

1. **플라이휠 검증에 필요한 것은 MVP 구현** — 참여 → 활동 데이터 → 가족 인사이트 → 재참여가 실제로 이어지는지 확인하는 데 필요한 전부.
2. **선결 조건(법 검토·실결제 게이트·데이터 축적·외부 계약)이 걸린 것은 발표 확장성 섹션** — 구현하지 않되, 설계에 이미 자리가 예약되어 있음을 서사로 소개한다.
3. "데모 상태"는 UI만 가짜라는 뜻이 아니다 — **금융 이행만 없고 서버 상태 관리(원장·entitlement·402 게이트)는 실구현**한다(서버 값 정본 불변식).

## 1. MVP 구현

### 1.1 즉시 실동작 — 데이터 축적 불필요 (단건 트랜잭션·상태 기반)

| 기능 | 비고 |
|---|---|
| 인증·가입(Apple+카카오)·약관 동의·역할 확정 | `POST /auth/session`, 가입 정보 5종 서버 저장 |
| 가족 연결(4자 초대 코드·폴링·형제자매 초대) | 동일 플로우 재사용 |
| **게임 미션 루프**: 6종 플레이 → 서버 판정 → 포인트 적립 | 판정은 attempt 단건 재계산 — 과거 기록 불필요. 멱등키·오프라인 큐 포함 |
| 체크인(기분·수면) | `(시니어, 날짜)` upsert, 최초 1회 +1,000P |
| 포인트 원장·용돈함·주간 목표·보상 규칙 설정 | append-only 원장 |
| **음성 응원 편지 전체 플로우** | 녹음→전사→안전 필터→우체통→재생→24h 파기. 분석 투입 금지 원칙이라 축적과 완전 무관 — **실동작 시연 하이라이트** |
| 음성 일기 수집·열람·삭제·공유 토글 | 단건 upsert + 인가. 분석은 §1.2 |
| 가족 선물하기·상점 구매(demo 이행)·아바타 꾸미기 | 단건 트랜잭션 + 서버 소유·수령인 검증 |
| 교육 콘텐츠 | 정적 콘텐츠 + 완료 보상 1회 |
| 알림함·알림 설정·디바이스 등록 | 푸시 발송 필터의 원천 |
| **케어 푸시 — 무활동 룰(2일 caution/3일 alert)** | 절대 룰이라 기준선 불필요 — 시드 하루치면 실발동 시연 가능 |
| 프리미엄 게이트(서버 402)·페이월 | entitlement 상태 판정 — 이력 무관 |
| 활동 달력·미션 히스토리·대시보드 1콜 | 첫날부터 동작, 쌓일수록 채워짐 |
| 똑똑이 리스킨(홈 인사말·대화형 체크인) | UI 작업만 — 서버 무관 |

### 1.2 시드 데이터로 시연 — 축적이 있어야 산출되는 인사이트

기능은 실구현하되, 시연 계정에 2~4주치 활동 데이터를 미리 넣어 보여준다.

| 기능 | 필요한 입력 |
|---|---|
| 추이 차트(F-2, 7/30일) | 시계열 자체 |
| 주간 리포트(F-3) | 1주치 + 사람 검수 |
| 참여 패턴(F-9)·케어 상태 이상 신호(정답률 하락·반응 지연) | 개인 기준선 — 무활동 룰과 달리 상대 룰 |
| 맞춤 추천·기피 게임 보너스 | 최근 14일 배정 대비 완료율 |
| 음성 일기 발화 특징 변화 추이 | 수 주치 시계열 |

### 1.3 데모 상태로 구현 — 금융 이행 없음, 서버 상태 관리는 실구현

| 기능 | 데모 방식 |
|---|---|
| 포인트 충전 | `status: "demo"` — 결제 없이 잔액 시뮬레이션, 원장 실기록 |
| 프리미엄 구독 결제 | demo 구독으로 티어 부여 — 402 게이트 플로우 실검증 |
| 출금 | `requested` 상태만, 계좌 마스킹 표시(실계좌 미수집) |
| 기프티콘·건강·제휴 상품 이행 | 전 상품 `fulfillment_type=demo` 서버 고정, 배송 주소 미수집 |

### 1.4 조건부 공개 — 보안 게이트 통과 후 한 묶음

음성 일기(S-20)·응원 편지(F-1/S-21)·우체통은 구현 완료 상태이나, **V-1~V-8·L-1~L-7 게이트 자체 감사 + 적대적 리뷰 통과 후 한 묶음으로 공개**한다(부분 출시 금지 — 보안 문서.md §4).

## 2. 발표 확장성 섹션

구현하지 않는다. 발표에서 "설계에 이미 자리가 예약되어 있다"는 서사로 소개한다(발표자료.md §6과 정합).

### 제품 확장 (B2C)

| 항목 | 예약된 슬롯 | 선결 조건 |
|---|---|---|
| 실결제·실출금·실발급 전환 | 데모 상태 스키마 구분, append-only 원장 | 자금 흐름 남용 방어 M-1~M-8(본인 명의 출금·한도·냉각·FDS·전금법/AML) |
| 무료 체험(trial) | `status: "trial"` enum 확정 | 기간·중복 방지 기준(A-7 ④) — 확정 전 페이월 미노출 |
| premium_ai 티어 | 구독 3티어 enum에 슬롯 예약 | 활동 데이터 축적 후 차별 기능 확정(A-7 ④) — 판매 금지·"준비 중" 표기 |
| 음성 일기 AI 정서 요약 | Gemini 배치 태스크 설계 완료·의도적 비활성 | 미학습 보장 약관 확인(A-7 ⑥) — 충족 시 켜기만 하면 됨 |
| 똑똑이 캐릭터 IP(굿즈·디지털 선물·캐릭터 구독) | 캐릭터가 경험의 중심에 배치됨 | 사업 확정 |
| 가족 통합 플랫폼(자녀 용돈 관리·가족 일정) | 가족 단위 데이터 구조 | 사업 확정 |
| 분담결제·N분의1 정산 | 충전 기여 기록(`charged_by`)까지 구현 | 실결제 확정 |

### 사업 확장 (B2B·B2G)

| 항목 | 예약된 슬롯 | 선결 조건 |
|---|---|---|
| 보험사 제휴·데이터 사업(비식별 집계 제공) | `third_party_research` 동의 코드 예약 | 비식별 적정성 평가 + 별도 동의 화면 + 처리방침 개정(보안 문서.md §5) |
| 기업 복지 제휴(B2B2C 구독 지원) | 구독 모델 위에 얹는 구조 | 계약 단위 확정 |
| 기관용 확장(복지관·요양기관·지자체) | 비진단 원칙 = 규제 적합성 자산 | 계약 단위 확정 |
| 제휴 상품 실이행·광고 표기 운영 | `partner` 카테고리·`partner_id` 필드 | 제휴 계약·정산 프로세스(A-7 ⑧) |
| 선택적 관계 인증(고한도 티어) | M-8에 티어 구조 설계 | 실출금 도입과 함께 |

## 3. 제거 — 어디에도 넣지 않음

걸음 기능 전체(S-9·홈 카드·주간 목표 걸음 조건), 같은 그림 찾기·소리 퀴즈·길 찾기, 휴대폰 OTP 로그인, 포인트 단가 조절 UI — 전부 제거 확정.
